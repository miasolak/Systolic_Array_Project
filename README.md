# Systolic Array 

This project implements a parameterized **systolic array architecture** for matrix multiplication in **Verilog HDL**. The design focuses on modular processing elements, coordinated control logic, and structured data flow, with verification through simulation.

A systolic array is a specialized architecture based on a network of cells called Processing Elements (PEs). The main idea of this system is that, within a certain number of clock cycles, the network of cells is filled with data representing the elements of the input matrices, after which the processing begins. In this way, parallel data processing is achieved, since multiplication and addition operations are performed simultaneously in the cells at every clock cycle. The number of clock cycles required for this type of processing is much smaller compared to other data-processing algorithms.


## Project Overview

The project includes:

- A parameterized systolic array for `N x M` matrix multiplication
- A modular **Processing Element (PE)**
- A **control logic** module
- A **Verilog testbench**
- A **Python script** for generating matrices and expected results

## Architecture

### `PE.v`
The basic cell of the array is the Processing Element (PE). It consists of three registers that store the data from each matrix as well as intermediate results. Within the PE, multiplication and addition operations are performed on these values.

### `systolic_array.v`
The top-level systolic array module represents the functional unit that connects all system components into a single architecture. It integrates:

- Processing Elements (PEs) that perform the basic arithmetic operations
- Control logic that manages the overall system behavior
- Internal interconnection signals required to connect all modules together

### `control_logic.v`
The control logic is implemented as a finite state machine (FSM) and consists of four phases. The FSM module contains four states that describe all operating phases of the systolic array: Idle, Load_AB, Wait, and Load_X.

#### `Idle`
In the `Idle` state, the system remains inactive and waits for the `ready_i` signal, which indicates that the input data is ready to be loaded. When `ready_i` goes high, the FSM transitions from `Idle` to `Load_AB`. Otherwise, it remains in `Idle`.

The `ready_i` signal is generated in the testbench and represents external synchronization between the data generator and the control logic.

#### `Load_AB`
In the `Load_AB` state, the values of matrices **A** and **B** are loaded into the array, and their propagation through the module begins. This is controlled by a counter that counts clock cycles.

When the counter reaches the smaller of the two dimensions, `min(M, N)`, the FSM transitions to the `Wait` state. If `M` and `N` are equal, the FSM moves directly to `Load_X`, since no additional propagation phase is required.

#### `Wait`
The `Wait` state is an additional phase used to complete propagation along the larger dimension of the array, either rows or columns. The FSM stays in this state for `|M - N|` clock cycles.

This ensures that all matrix elements reach their correct positions in the array before computation begins.

#### `Load_X`
In the `Load_X` state, result processing is performed. The values are multiplied, then added to the previous intermediate result, and the updated result is propagated vertically downward.

In this state, the counter runs for `N` clock cycles, after which the `ready_i` signal is checked again:

- If `ready_i` is high, the FSM returns to `Load_AB`, meaning that new input data is ready
- If `ready_i` is low, the FSM transitions to `Idle`
- Otherwise, it remains in `Load_X`

### `systolic_array_tb.v`
The testbench simulates the operation of the entire system. Its role is to:

- Generate all required input signals
- Provide synchronization for the design
- Load data from files generated by the Python script
- Compare the results produced by the systolic array with the expected values
At the beginning of the file, the array dimensions and data width are defined through the parameters `N`, `M`, and `DATA_WIDTH`.

### `matricesABC.py`
A Python script was developed to automatically generate an arbitrary number of matrix pairs **A** and **B**, with different values and dimensions.

The main parameters are:

- `M` – number of rows in the array
- `N` – number of columns in the array
- `K` – number of matrix pairs to be generated and tested

## How It Works

- Matrix **A** values move horizontally
- Matrix **B** values move vertically
- Each PE performs multiply-accumulate operations
- Final matrix results are produced after several clock cycles

## Features

### Parameterized Array Dimensions
The entire module is parameterized so that the dimensions of the array can be changed as needed. The number of columns is defined by **M**, while the number of rows is defined by **N**. These two values do not have to be equal, meaning that the array does not need to be square.


The same hardware implementation can therefore support different configurations, ranging from small experimental arrays to larger systolic networks. In addition to array dimensions, the number of matrix pairs entering the system is also parameterized.

### Data Width Parameterization
Besides array dimensions, the data width is also parameterized. It is defined independently of the matrix size, but matrices **A** and **B** must have the same data width to ensure correct arithmetic processing.


### Output Validity Signaling
After the input data is loaded, the next phase is processing. Valid output results appear only after a certain number of clock cycles. Therefore, a dedicated signal is used to indicate when the output values are valid.
